## Overview

The Resume Audit Engine is a pluggable engine that performs comprehensive audit and analysis of resume documents. It generates structured signals about resume quality, ATS compatibility, content analysis, and improvement recommendations.

**Engine Type**: Signal Processing Engine  
**Base Class**: `BaseEngine` (from `gateai.gateai.engines.base`)  
**Namespace**: Reserved for future implementation

## Input Schema

**Pydantic Model**: `ResumeAuditInput`

```python
class ResumeAuditInput(BaseModel):
    """Input schema for Resume Audit Engine"""
    
    # Required fields
    resume_id: str = Field(..., description="Resume identifier in the system")
    resume_text: str = Field(..., min_length=1, description="Extracted text content from resume")
    user_id: str = Field(..., description="User identifier who owns the resume")
    
    # Optional context fields
    target_job_title: Optional[str] = Field(None, description="Target job title for context-aware analysis")
    target_industry: Optional[str] = Field(None, description="Target industry for context-aware analysis")
    target_keywords: Optional[List[str]] = Field(default_factory=list, description="Target keywords to check for")
    
    # Analysis configuration
    analysis_depth: Literal["basic", "standard", "comprehensive"] = Field(
        default="standard",
        description="Depth of analysis to perform"
    )
    include_recommendations: bool = Field(
        default=True,
        description="Whether to include improvement recommendations"
    )
    include_ats_compatibility: bool = Field(
        default=True,
        description="Whether to perform ATS compatibility analysis"
    )
    
    # Metadata
    resume_file_type: Optional[str] = Field(None, description="Original file type (pdf, docx, etc.)")
    resume_file_size: Optional[int] = Field(None, ge=0, description="File size in bytes")
    upload_timestamp: Optional[datetime] = Field(None, description="When resume was uploaded")
```

**Validation Rules**:
- `resume_id` must be non-empty string
- `resume_text` must contain at least 1 character
- `user_id` must be non-empty string
- `target_keywords` must be a list of strings (can be empty)
- `analysis_depth` must be one of: "basic", "standard", "comprehensive"
- `resume_file_size` must be non-negative if provided

## Signal Types

The Resume Audit Engine generates structured signals of the following types:

### 1. Structure Issues (`structure_issue`)
**Purpose**: Identify problems with resume structure and formatting

**Severity Levels**:
- `critical`: Missing essential sections (e.g., contact information, work experience)
- `high`: Major structural problems (e.g., inconsistent formatting, poor organization)
- `medium`: Moderate structural issues (e.g., section ordering, spacing)
- `low`: Minor formatting inconsistencies

**Example Signals**:
- Missing contact information section
- Inconsistent date formatting
- Poor section organization
- Missing summary/objective section

### 2. Content Issues (`content_issue`)
**Purpose**: Identify problems with resume content quality

**Severity Levels**:
- `critical`: Critical content problems (e.g., no work experience, missing education)
- `high`: Major content gaps (e.g., insufficient detail, vague descriptions)
- `medium`: Content quality issues (e.g., weak action verbs, generic descriptions)
- `low`: Minor content improvements needed

**Example Signals**:
- Work experience descriptions lack quantifiable achievements
- Generic job descriptions without specific accomplishments
- Missing relevant experience details
- Weak action verbs used

### 3. Keyword Missing (`keyword_missing`)
**Purpose**: Identify missing important keywords

**Severity Levels**:
- `critical`: Missing essential industry keywords
- `high`: Missing important job-specific keywords
- `medium`: Missing preferred keywords
- `low`: Missing optional keywords

**Example Signals**:
- Missing target job title keywords
- Missing industry-standard terminology
- Missing technical skill keywords
- Missing soft skill keywords

### 4. ATS Incompatibility (`ats_incompatibility`)
**Purpose**: Identify ATS compatibility issues

**Severity Levels**:
- `critical`: Major ATS blockers (e.g., images, complex formatting)
- `high`: Significant ATS issues (e.g., non-standard fonts, tables)
- `medium`: Moderate ATS concerns (e.g., special characters, formatting)
- `low`: Minor ATS optimization opportunities

**Example Signals**:
- Resume contains images or graphics
- Complex table formatting detected
- Non-standard fonts used
- Special characters that may cause parsing issues

### 5. Skill Gaps (`skill_gap`)
**Purpose**: Identify missing skills relevant to target role

**Severity Levels**:
- `critical`: Missing critical required skills
- `high`: Missing important skills
- `medium`: Missing preferred skills
- `low`: Missing nice-to-have skills

**Example Signals**:
- Missing required technical skills for target role
- Missing industry-standard tools/technologies
- Missing soft skills relevant to role
- Skill level mismatch

### 6. Formatting Issues (`formatting_issue`)
**Purpose**: Identify formatting and presentation problems

**Severity Levels**:
- `high`: Major formatting problems
- `medium`: Moderate formatting issues
- `low`: Minor formatting improvements

**Example Signals**:
- Inconsistent spacing
- Font size inconsistencies
- Alignment issues
- Page break problems

### 7. Strengths (`strength`)
**Purpose**: Highlight resume strengths and positive aspects

**Severity Levels**:
- `info`: Positive aspects to highlight

**Example Signals**:
- Strong quantifiable achievements
- Relevant certifications
- Industry-relevant experience
- Well-structured sections

### 8. Recommendations (`recommendation`)
**Purpose**: Provide actionable improvement recommendations

**Severity Levels**:
- `high`: High-priority recommendations
- `medium`: Medium-priority recommendations
- `low`: Low-priority recommendations

**Example Signals**:
- Add specific keywords
- Improve section descriptions
- Add missing certifications
- Optimize for ATS

## DecisionSlot Anchoring Logic

**Rule 15: SIGNAL INTEGRITY** - Every engine output MUST be anchored to a DecisionSlot ID.

### DecisionSlot Creation/Retrieval

1. **Input Validation**: The engine receives `decision_slot_id` as a required parameter in the `run()` method.

2. **DecisionSlot Requirements**:
   - The `decision_slot_id` must reference an existing DecisionSlot in the system
   - The DecisionSlot must be in a state that allows signal attachment
   - The DecisionSlot should be associated with the user who owns the resume

3. **Signal Anchoring**:
   - All output signals MUST include the `decision_slot_id` in the output
   - The `decision_slot_id` is included in the top-level `ResumeAuditOutput` object
   - Each individual signal in the `signals` list is implicitly anchored to the same DecisionSlot
   - No signals are generated without a valid DecisionSlot ID

4. **DecisionSlot State Management**:
   - The engine does NOT modify DecisionSlot state directly (Rule 6: No Direct DB Access)
   - The engine only references the DecisionSlot ID for signal integrity
   - State management is handled by the kernel (decision_slots layer)

5. **Error Handling**:
   - If `decision_slot_id` is invalid or missing, the engine MUST raise a `ValueError`
   - Fallback logic MUST still anchor to the provided `decision_slot_id` (even if invalid)
   - The kernel layer is responsible for validating DecisionSlot existence

### Example Anchoring Flow

```
1. User uploads resume → Kernel creates DecisionSlot (ID: "ds_12345")
2. Kernel calls engine.run(raw_input, decision_slot_id="ds_12345")
3. Engine processes resume and generates signals
4. Engine returns output with decision_slot_id="ds_12345" in all signals
5. Kernel persists signals to ats_signals layer, anchored to DecisionSlot "ds_12345"
```

## Engine Interface

### Required Methods (from BaseEngine)

1. **`run(raw_input: dict, decision_slot_id: str) -> dict`**
   - Validates input against `ResumeAuditInput` schema
   - Performs resume audit analysis
   - Generates structured signals
   - Returns output conforming to `ResumeAuditOutput` schema
   - MUST anchor all output to `decision_slot_id` (Rule 15)

2. **`fallback_logic(error: Exception, raw_input: dict, decision_slot_id: str) -> dict`**
   - Called when `run()` fails
   - Returns minimal valid output with error information
   - MUST still conform to `ResumeAuditOutput` schema
   - MUST anchor to `decision_slot_id` (Rule 15)
   - Should include error details in signals

3. **`trace_metadata() -> Dict[str, Any]`**
   - Returns execution metadata:
     - `latency_ms`: int - Processing time in milliseconds
     - `tokens_used`: Optional[int] - LLM tokens consumed (if applicable)
     - `model_version`: Optional[str] - AI model version used (if applicable)

### Schema Definitions

- **`input_schema`**: `ResumeAuditInput` (Pydantic model)
- **`output_schema`**: `ResumeAuditOutput` (Pydantic model)

---

## Fallback Logic Specification

When `run()` fails, `fallback_logic()` MUST return a valid `ResumeAuditOutput` with:

1. **Minimal Valid Output**:
   - `decision_slot_id`: Use provided ID (even if invalid)
   - `overall_score`: 0.0 (indicates failure)
   - `score_category`: "poor"
   - All component scores: 0.0
   - `signals`: Include at least one signal with:
     - `signal_type`: "structure_issue"
     - `severity`: "critical"
     - `message`: Error message describing the failure
     - `details`: {"error_type": str(error.__class__.__name__), "error_message": str(error)}

2. **Metadata**:
   - `engine_version`: Current engine version
   - `timestamp`: Current timestamp
   - `processing_time_seconds`: 0.0
   - `confidence_score`: 0.0
   - `analysis_depth_used`: "basic" (fallback mode)

3. **Empty Collections**:
   - All lists (keywords, skills, etc.) should be empty
   - `recommendations`: None

**Rule 10: Traceability**

## Integration Points

### Kernel Integration

1. **Input Source**: Kernel receives resume data and creates DecisionSlot
2. **Engine Execution**: Kernel calls `engine.run(raw_input, decision_slot_id)`
3. **Output Processing**: Kernel validates output against `ResumeAuditOutput` schema
4. **Signal Persistence**: Kernel persists signals to `ats_signals` layer
5. **DecisionSlot Update**: Kernel updates DecisionSlot state (engine does not)

### External Service Integration

- **No Direct DB Access** (Rule 6): Engine does not access database directly
- **No Direct API Calls**: Engine does not make external API calls directly
- **Service Isolation**: All external services accessed through kernel-provided interfaces

---

## Error Handling

### Input Validation Errors

- **Invalid `decision_slot_id`**: Raise `ValueError` with message "Invalid decision_slot_id"
- **Invalid input schema**: Raise `ValidationError` (Pydantic) with validation details
- **Missing required fields**: Raise `ValueError` with specific field names

### Processing Errors

- **Resume parsing failure**: Trigger `fallback_logic()` with parsing error
- **Analysis failure**: Trigger `fallback_logic()` with analysis error
- **External service failure**: Trigger `fallback_logic()` with service error

### Output Validation Errors

- **Output schema violation**: Raise `ValidationError` before returning
- **Missing `decision_slot_id`**: Raise `ValueError` (Rule 15 violation)
